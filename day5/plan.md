# 综合实验：高保真银行排队调度仿真系统（C语言版）

## 一、 项目概述

本实验旨在构建一个**非平稳、多画像、动态调度**的银行排队离散事件仿真系统。程序需使用 C 语言实现，摒弃传统的“均匀到达+单一业务”简化模型，转而采用**分时段泊松过程、客户分层画像、精细化业务矩阵及弹性窗口策略**。系统需在满足基础教学知识点（结构体、队列、文件读写、随机模拟）的前提下，真实还原银行网点的复杂运营场景，并输出多维度统计日志。

## 二、 核心功能需求

### 1. 客户到达模型（非平稳 & 突发）

- **分时段到达率配置**：摒弃全天均匀随机。需支持通过配置文件或宏定义设置不同时段的平均到达率 λ(t)（单位：人/分钟）。
  - *示例配置*：08:00-09:00 (高峰 λ=8)、11:30-13:30 (午休低谷 λ=2)、16:00-17:00 (晚高峰 λ=7)。
- **客户画像分层生成**：到达客户需按概率随机赋予画像标签，不同画像影响后续业务与优先级：
  - `VIP_CLIENT` (10%)：高优先级，专属业务概率高。
  - `CORPORATE` (15%)：对公客户，办理时长长，仅特定窗口可受理。
  - `ELDERLY` (20%)：老年客户，优先窗口，但业务耗时波动大。
  - `NORMAL` (55%)：普通个人客户。
- **批量/突发流量注入**：仿真过程中随机时刻瞬时插入一批客户（如15~25人），模拟真实世界的脉冲式流量。

### 2. 业务办理精细化模型

- **业务类型矩阵**：每位客户的业务类型由“客户画像”决定概率分布。每种业务对应独立的耗时分布参数（建议采用对数正态分布或伽马分布，而非简单均匀分布）：
  - 现金存取、转账汇款、开户销户、理财咨询、对公结算等至少5类。
- **柜员能力差异系数**：每个窗口绑定一个“熟练度系数” k ∈ [0.8, 1.5]。
  - 实际办理时长 = `基准业务时长 × k`。
  - 新员工 (k=1.5) vs 资深员工 (k=0.8)。
- **异常中断模拟**：办理过程中有低概率（如2%）触发异常（系统故障/卡纸/资料缺失），导致当前服务暂停 T_err 时间或强制结束并要求客户重新排队。

### 3. 智能排队与动态调度策略

- **多级优先级队列**：
  - 每个窗口维护一个逻辑队列（或全局优先队列），排序权重 = `基础FIFO序 + 画像优先级权重 + 预约标识`。
  - VIP/老弱病残在同等条件下优先叫号；预约客户在预约时间点前后享有插队权。
- **弹性/潮汐窗口机制**：
  - **自动切换**：当普通队列长度 < 阈值 且 对公队列积压 > 阈值时，空闲普通窗口自动切换为对公窗口。
  - **潮汐控制**：指定窗口仅在高峰时段（如08:00-10:00）开启，其余时间关闭。
- **弃号与重排行为（Balking & Jockeying）**：
  - **Balking**：客户到达时若发现最短队列长度 > 10人，以一定概率直接离开不排队。
  - **Jockeying**：排队中若相邻队列长度比当前少2人以上，以一定概率换队。
  - **过号重排**：叫号后未在超时窗口内响应，视为过号，重新插入队尾（优先级降级）。
- **营业时间约束**：17:00 停止取号/新客户进入；20:00 强制关门，未完成服务的客户记录为“未办结”并终止仿真。

### 4. 数据统计与持久化（实验硬性指标）

- **实时控制台输出**：关键事件（客户到达、开始服务、完成、弃号、窗口切换）需带时间戳打印。
- **多维统计指标**：
  - 全局：平均等待时长、最大排队长度、客户流失率（Balking比例）、业务完成率。
  - 分画像：各类客户平均等待时长、满意度估算。
  - 分窗口：各窗口接待总量、平均办理时长、空闲率、异常中断次数。
- **日志文件写入**：所有统计数据及仿真过程摘要必须写入 `simulation_log.txt`，格式清晰可读。

## 三、 技术实现约束（C语言）

1. **数据结构**：必须使用 `struct` 定义客户节点（含画像、业务类型、到达时间、优先级、状态标记）、窗口状态、事件节点。
2. **队列实现**：使用数组或链表模拟优先队列/普通队列，禁止直接使用STL/C++容器。
3. **随机数引擎**：自行封装随机数生成器，支持均匀分布、指数分布、对数正态分布的采样（用于到达间隔和业务时长）。
4. **仿真驱动**：采用**离散事件模拟（DES）** 框架，基于事件表（Event List）推进时间，而非固定时间步长轮询。
5. **文件操作**：使用标准 C IO (`fopen`, `fprintf`, `fclose`) 完成日志写入。
6. **代码规范**：模块化设计（到达模块、服务模块、调度模块、统计模块分离），注释完整。

## 四、 验收标准

| 维度 | 要求 |
| :--- | :--- |
| **功能完整性** | 覆盖上述所有核心功能点，无编译错误 |
| **模型真实性** | 到达率随时间变化明显；VIP确实比普通客户等待短；突发流量导致队列骤增 |
| **数据正确性** | 各窗口服务量均衡度偏差<15%；统计指标数值合理（如平均等待不应为负） |
| **日志规范性** | 日志文件包含完整的仿真摘要与分项统计，格式对齐 |
| **代码质量** | 结构体定义清晰，队列操作封装良好，随机数生成独立可测试 |

## 五、 提示与建议

- **事件表设计**：建议定义事件类型枚举 `{ARRIVAL, BULK_ARRIVAL, START_SERVICE, END_SERVICE, WINDOW_SWITCH, NO_RESPONSE_TIMEOUT, SHUTDOWN}`，事件表按时间戳排序的最小堆或有序链表是DES的核心。
- **配置外部化**：虽然要求C语言，但建议用宏或简单的配置解析函数管理时段、概率、系数等参数，避免硬编码。
- **调试策略**：先关闭所有高级特性（优先级、弹性窗口、异常），跑通基础FIFO+分时到达；再逐步叠加复杂逻辑。
- **验证方法**：固定随机种子进行可复现测试；极端参数测试（如λ=0或λ=100）检验边界处理。
